# Business Events Collector
Сервис для обработки файлов с бизнес-событиями и отправки событий в систему аналитики.

## Описание
Основная задача сервиса — чтение файлов событий, десериализация записей в объекты и отправка событий в очередь сообщений.
Сервис запускается на конечном устройстве (например, кассовом аппарате) и обрабатывает файлы с событиями операций. 
Файлы содержат данные о чеках, которые были оплачены или отменены.

Проект является частью системы аналитики бизнес-событий и включает два компонента:
- **Collector** — запускает обработку файлов с событиями, используя Processor для параллельной обработки и отправки данных.
- **Generator** — инструмент для создания тестовых файлов с событиями.

## Технологии  
Проект разработан с использованием:
- **.NET 8** — платформа для разработки и запуска сервиса  
- **Docker / Docker Compose** — контейнеризация сервисов  
- **PostgreSQL** — база данных для хранения событий и логов
- **Moq / XUnit** — библиотеки для модульного тестирования  
- **Serilog** — логирование работы сервиса  
- **Bogus** — генерация тестовых данных

## Структура решения
Решение состоит из нескольких проектов:

###  dnaborshchikova_github.Bea.Collector.App
Консольное приложение для разового запуска обработки файлов из заданной директории. 
Запускает обработку файлов с событиями, используя Processor для параллельной обработки и отправки данных. 
Читает один файл и завершает работу.

### dnaborshchikova_github.Bea.Collector.WorkerService
Сервисное приложение, которое запускает обработку файлов с событиями через Processor.
Работает в двух режимах:
- Однократная обработка файлов (например, при запуске в Docker-контейнере)  
- Периодический запуск (раз в 24 часа) как воркер-сервис ОС.

### dnaborshchikova_github.Bea.Collector.Core
Содержит базовые модели и интерфейсы проекта.

### dnaborshchikova_github.Bea.Collector.DataAccess
Проект содержит репозиторий и контекст базы данных. Отвечает за инициализацию базы данных и сохранение событий.

### dnaborshchikova_github.Bea.Collector.Parser
Модуль разбора файла событий и десериализации данных в объекты.

### dnaborshchikova_github.Bea.Collector.Processor
Центральный компонент обработки событий.  
- Получает файлы с событиями от App или WorkerService.  
- Разбирает файлы и преобразует записи в объекты событий.  
- Делит события на части для параллельной обработки (потоки или задачи в зависимости от настроек).  
- Отправляет события в базу данных или в очередь сообщений.  
- Логирует результаты обработки каждого файла.

### dnaborshchikova_github.Bea.Collector.Sender
Отправляет события в очередь сообщений и базу данных.

### dnaborshchikova_github.Bea.Collector.Tests
Модульные тесты для проекта.

### dnaborshchikova_github.Bea.Generator
Генератор тестовых файлов с событиями для проверки работы системы.

## Формат файлов с событиями
Сервис обрабатывает CSV-файлы с событиями операций. Поддерживаются два типа событий:

### Событие оплаты чека (`bill_payed`)
| Поле | Описание |
|-----|-----|
| OperationDateTime | дата и время операции |
| UserId | GUID кассира |
| EventType | `bill_payed` |
| Amount | сумма чека |
| Number | номер чека (6-значный, уникален в течение дня) |
| Id | GUID чека |
| BuyerId | GUID покупателя (необязательный) |

### Событие отмены чека (`bill_canceled`)
| Поле | Описание |
|-----|-----|
| OperationDateTime | дата и время операции |
| UserId | GUID кассира |
| EventType | `bill_canceled` |
| Amount | общая сумма чека |
| CancelAmount | сумма отмены |
| Number | номер чека (6-значный, уникален в течение дня) |
| Id | GUID чека |

> Parser преобразует строки файла в внутренние события, которые отправляются в очередь сообщений или базу данных.

## Режимы запуска
Сервис поддерживает конфигурацию через `appsettings.json` или через Docker environment. 
Настройки определяют режим работы сервиса, способ обработки файлов и запуск генератора тестовых данных.

### ProcessingSettings

- `ThreadCount` — количество потоков или задач для параллельной обработки событий (по умолчанию 5–15).  
- `ProcessType` — способ обработки: `Thread` для потоков или `Task` для задач.  
- `FilePath` — путь к CSV-файлу для обработки.  
- `InputFolder` — папка для мониторинга новых файлов (актуально для WorkerService).  
- `GeneratorRunAsProcess` — запуск генератора тестовых файлов как отдельного процесса (`true` / `false`).  
- `GenerateFile` — генерация тестового файла перед обработкой (`true` / `false`). 
- `RunMode` — режим работы сервиса:
	- `OneTime` — однократная обработка файла и завершение (например, запуск в Docker)
	- `ScheduledService` — Windows-сервис, который периодически (по расписанию, например раз в 24 часа) проверяет папку и обрабатывает новые файлы

### GeneratorSettings

- `FileFormat` — формат создаваемого тестового файла (`csv`).  
- `PaidBillEventCount` — количество событий оплаты чека, которое будет сгенерировано в файле.  
- `CancelledBillEventCount` — количество событий отмены чека, которое будет сгенерировано в файле.  

> Processor делит файл на диапазоны по датам и обрабатывает события параллельно в соответствии с настройками. Настройки можно изменять напрямую в `appsettings.json` или через Docker environment.

## Запуск проекта  
  
1. Склонировать репозиторий: `git clone <repo>`
2. Перейти в директорию проекта: `cd <repo-folder>`  
3. Запустить сервис через Docker: `docker-compose up`
> Примечание: Базовые настройки уже прописаны в `appsettings.json`, проект можно сразу запускать без изменения конфигурации.

## Результат работы

После запуска сервис:

1. Читает CSV-файл с событиями из `InputFolder`.  
2. Парсит записи и преобразует их в объекты событий.  
3. Делит события на диапазоны и обрабатывает их параллельно (в потоках или задачах).  
4. Отправляет события в очередь сообщений и/или сохраняет их в базу данных.  
5. Логирует результаты: количество обработанных событий, диапазоны и успешность отправки.  


Логи работы сервиса можно посмотреть в консоли или через Docker:  
`docker-compose logs -f`  
  
### Данные в базе  
  
После обработки события сохраняются в базе данных.  
Для проверки результатов можно посмотреть следующие таблицы:  
  
- **SendEvents** — содержит обработанные бизнес-события.  
- **SendEventLogs** — содержит информацию о запуске обработки файла и результате отправки событий.
